# 结构迁移映射与实施顺序

## 1. 迁移原则

当前工程只保留参考结构中对 OpenIM 有实际价值的现行分支。页面层级和阅读路径可以参考 Sendbird，但接口、对象模型、权限、术语、生命周期和平台能力必须以 OpenIM 真实实现为准。

迁移每一页时执行三步：

1. 判断 OpenIM 是否存在对应能力。
2. 使用 OpenIM 的代码、接口规范、测试和产品定义重写正文。
3. 不存在的能力直接删除、合并或指向替代路径，不为保持页面数量而虚构内容。

`sourcePath` 仅用于追踪原结构来源，不在页面 UI 中展示。

## 2. 当前产品映射

| 结构分支                        | OpenIM 目标内容        | 处理方式                               |
| ------------------------------- | ---------------------- | -------------------------------------- |
| `/docs/chat`                    | OpenIM Chat 总入口     | 产品定位、最短成功路径和主要入口       |
| `/docs/chat/sdk/v4/**`          | OpenIM Client SDK 指南 | 按各平台真实包、初始化方式和能力重写   |
| `/docs/chat/platform-api/v3/**` | OpenIM Server REST API | 按当前鉴权、接口、错误码和权限模型重写 |

不再迁移 UIKit、SDK v3、Legacy API 和手写 SDK Reference 页面。

## 3. 平台决策

当前槽位包含 iOS、Android、Flutter、uni-app、WASM、Electron、小程序、React Native 和 Unity。开始编写某个平台前，建立真实支持矩阵：

| 状态        | 页面处理                                     |
| ----------- | -------------------------------------------- |
| 正式支持    | 编写真实安装、初始化、核心流程和兼容范围     |
| Beta/实验性 | 使用 `status: draft`，明确稳定性和限制       |
| 社区维护    | 明确维护仓库、负责人和支持边界               |
| 不支持      | 删除对应上下文或合并到替代平台，不保留假页面 |

如果 OpenIM 需要增加 Electron、React Native 等独立平台，应按真实产品边界增加，而不是受参考站平台列表限制。

## 4. 推荐实施波次

### Wave 0：范围、品牌与规范

- 确认正式支持的平台和当前版本。
- 替换 Logo、颜色、域名和页头入口。
- 建立术语表、代码示例规范和版本政策。
- 确认哪些参考栏目需要删除或合并。

### Wave 1：最短成功路径

- Chat 首页。
- Server 部署和准备说明。
- 主要 SDK 的安装、初始化、登录、会话和发送首条消息。
- Server API 的鉴权、请求约定、响应格式和错误码。

### Wave 2：核心功能指南

- 用户与关系链。
- 群组、会话与成员。
- 文本、图片、文件、语音等消息。
- 已读回执、输入状态、撤回、引用、搜索。
- 推送、离线同步与重连。

### Wave 3：管理与运营能力

- Webhook/回调。
- 管理员、审核、封禁和风控。
- 数据导出、日志、监控与限流。
- 权限、安全和多租户边界。

### Wave 4：高级集成

- 性能、弱网、规模化和故障排查。
- Electron、React Native 或其他真实支持平台。
- 示例应用和业务集成模式；不重新引入空泛 UIKit 分支。

### Wave 5：Reference 自动化

- 从 SDK 源码或类型定义生成客户端 API Reference。
- 从 OpenAPI 生成 Server API Schema 与端点参考。
- 用稳定链接把手写指南连接到生成内容。
- 保证生成物不与 MDX 手写内容形成冲突事实源。

## 5. 页面级决策表

每个页面开始编写前建议记录：

| 项目            | 示例                                   |
| --------------- | -------------------------------------- |
| 结构路由        | `/docs/chat/sdk/v4/wasm/...`           |
| OpenIM 对应能力 | 会话列表                               |
| 真实来源        | SDK 代码、接口规范、测试用例、产品说明 |
| 内容负责人      | WASM SDK 团队                          |
| 技术审核人      | SDK Maintainer                         |
| 目标状态        | `published`                            |
| 删除/合并/替代  | 无或具体目标 URL                       |

## 6. 迁移进度

Frontmatter 的 `status` 是唯一迁移状态源：

```bash
npm run content:status
npm run content:status -- --status scaffold --limit 100
```

按产品、上下文和模板的统计可直接用于任务分配和验收。

## 7. 发布策略

不建议首次上线公开所有脚手架页。正式发布时可：

- Sitemap 只输出 `published` 页面。
- 对 `scaffold` 和 `draft` 页面增加 `noindex` 或从导航隐藏。
- 按 SDK 平台或 Server API 模块分批开放。
- 对被合并的页面配置显式重定向。

当前工程为了方便内容迁移，仍允许所有保留槽位被访问。
