## Overview

Migrate Sendbird-style Web chat code by replacing app/channel/user/message primitives with OpenIM users, conversations, groups, messages, and event callbacks.

This guide uses public OpenIM WASM SDK signatures and assumes the SDK version is compatible with OpenIM Server.

## OpenIM model

OpenIM does not implement Sendbird collections, open channels, polls, scheduled messages, or reactions as equivalent client SDK objects. Keep those as backend or custom-message features during migration.

Support status: public WASM SDK APIs cover this workflow.

Key concepts to keep straight:

- `userID` identifies an OpenIM user.
- `conversationID` identifies a local conversation row.
- `recvID` sends a direct message to one user.
- `groupID` sends to or operates on a group conversation.
- `operationID` is optional on most methods and is useful for tracing client/server logs.

## Prerequisites

- Install an OpenIM WASM SDK version that is compatible with OpenIM Server.
- Serve `openIM.wasm`, `sql-wasm.wasm`, and `wasm_exec.js` from the browser app public path.
- Create one SDK instance with `getSDK()` and call `login()` with `userID`, token, `platformID: 5`, `apiAddr`, and `wsAddr`.

## API signatures

Relevant public signatures:

```ts
function getSDK(config?: WasmPathConfig): SDK;
login: (params: InitAndLoginConfig, operationID?: string) => Promise<any>;
getConversationIDBySessionType: (data: GetOneConversationParams, operationID?: string) =>
  Promise<WsResponse<string>>;
createGroup: (data: CreateGroupParams, operationID?: string) => Promise<WsResponse<GroupItem>>;
sendMessage: (params: SendMsgParams, operationID?: string) => Promise<WsResponse<MessageItem>>;
getAdvancedHistoryMessageList: (params: GetAdvancedHistoryMsgParams, operationID?: string) =>
  Promise<WsResponse<AdvancedGetMessageResult>>;
searchLocalMessages: (data: SearchLocalParams, operationID?: string) =>
  Promise<WsResponse<SearchMessageResult>>;
```

Relevant events:

```ts
on<E extends CbEvents>(event: E, fn: Cbfn<E>): this;
off<E extends CbEvents>(event: E, fn: Cbfn<E>): this | undefined;
CbEvents.OnRecvNewMessages
CbEvents.OnConversationChanged
CbEvents.OnNewConversation
CbEvents.OnTotalUnreadMessageCountChanged
```

Response and event payloads use these common shapes:

```ts
type WsResponse<T> = {
  event: string;
  errCode: number;
  errMsg: string;
  data: T;
  operationID: string;
};
type WSEvent<T = unknown> = {
  event: CbEvents;
  data: T;
  errCode: number;
  errMsg: string;
  operationID: string;
};
```

## Implementation

Migrate by concept, not by method name.

```ts
const OpenIM = getSDK({
  coreWasmPath: '/openIM.wasm',
  sqlWasmPath: '/sql-wasm.wasm',
});
await OpenIM.login({ userID, token, platformID: 5, apiAddr, wsAddr });

const { data: conversations } = await OpenIM.getAllConversationList();
const { data: message } = await OpenIM.createTextMessage('Migrated hello');
await OpenIM.sendMessage({ recvID, groupID: '', message });
```

Replace Sendbird open channels, polls, scheduled messages, reactions, and collection APIs with explicit OpenIM group/conversation/message APIs plus backend services where needed.

## Returned data and events

The SDK response `data` field contains the updated OpenIM object or operation result. Persist IDs such as `conversationID`, `groupID`, and `clientMsgID` in your application state.

Watch these callbacks while testing: `CbEvents.OnRecvNewMessages`, `CbEvents.OnConversationChanged`, `CbEvents.OnNewConversation`, `CbEvents.OnTotalUnreadMessageCountChanged`.

Always log `operationID` with failures. It is the fastest way to match browser behavior with OpenIM server logs.

## Verify the result

- Confirm `login()` succeeds and `CbEvents.OnConnectSuccess` fires before calling the task API.
- Confirm the OpenIM call resolves and the returned data or follow-up SDK event matches the expected state.
- Confirm the expected SDK event fires in another signed-in browser session when the action should be visible to other users.

## Troubleshooting

| Case                           | Cause                                                                 | Handling                                                                                               |
| ------------------------------ | --------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| WASM asset requests return 404 | The SDK assets were not copied to the app public path.                | Copy `openIM.wasm`, `sql-wasm.wasm`, and `wasm_exec.js` from the npm package assets directory.         |
| The client never connects      | `apiAddr`, `wsAddr`, token, platform ID, or network routing is wrong. | Register connection events before `login()` and test the API and WebSocket URLs from the same browser. |
| The SDK call rejects | The server rejected the request or the current user lacks permission. | Handle the error in `catch`, log the rejected error with `operationID`, and verify group role, token, and target IDs. |
