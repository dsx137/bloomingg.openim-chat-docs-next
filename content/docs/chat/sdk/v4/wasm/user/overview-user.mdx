## Overview

Use user APIs for current-user profile, application user lookup, online status, friendship, and blacklist views.

Conversation display names belong to conversation data. Use `ConversationItem.showName` for conversation lists and direct-chat headers instead of filling those titles with `getUsersInfo()`.

This guide uses public OpenIM WASM SDK signatures and assumes the SDK version is compatible with OpenIM Server.

## OpenIM model

OpenIM separates account profile, friendship, blacklist, online status, and group membership. Avoid treating every SDK user call as an application-wide directory query.

Support status: public WASM SDK APIs cover this workflow.

Key concepts to keep straight:

- `userID` identifies an OpenIM user.
- `conversationID` identifies a local conversation row.
- `recvID` sends a direct message to one user.
- `groupID` sends to or operates on a group conversation.
- `operationID` is optional on most methods and is useful for tracing client/server logs.

## Prerequisites

- Install an OpenIM WASM SDK version that is compatible with OpenIM Server.
- Serve `openIM.wasm`, `sql-wasm.wasm`, and `wasm_exec.js` from the browser app public path.
- Create one SDK instance with `getSDK()` and call `login()` with `userID`, token, `platformID: 5`, `apiAddr`, and `wsAddr`.

## API signatures

Relevant public signatures:

```ts
function getSDK(config?: WasmPathConfig): SDK;
getSelfUserInfo: (operationID?: string) => Promise<WsResponse<SelfUserInfo>>;
setSelfInfo: <T>(data: PartialUserItem, operationID?: string) => Promise<WsResponse<T>>;
getUsersInfo: (data: string[], operationID?: string) => Promise<WsResponse<PublicUserItem[]>>;
subscribeUsersStatus: (data: string[], operationID?: string) =>
  Promise<WsResponse<UserOnlineState[]>>;
getFriendListPage: (data: OffsetParams & { filterBlack?: boolean }, operationID?: string) =>
  Promise<WsResponse<FriendUserItem[]>>;
addBlack: <T>(data: AddBlackParams, operationID?: string) => Promise<WsResponse<T>>;
```

Relevant events:

```ts
on<E extends CbEvents>(event: E, fn: Cbfn<E>): this;
off<E extends CbEvents>(event: E, fn: Cbfn<E>): this | undefined;
CbEvents.OnSelfInfoUpdated
CbEvents.OnUserStatusChanged
CbEvents.OnFriendAdded
CbEvents.OnFriendDeleted
CbEvents.OnBlackAdded
CbEvents.OnBlackDeleted
```

Response and event payloads use these common shapes:

```ts
type WsResponse<T> = {
  event: string;
  errCode: number;
  errMsg: string;
  data: T;
  operationID: string;
};
type WSEvent<T = unknown> = {
  event: CbEvents;
  data: T;
  errCode: number;
  errMsg: string;
  operationID: string;
};
```

## Implementation

Load user state by scope.

```ts
const { data: me } = await OpenIM.getSelfUserInfo();
const usersResponse = await OpenIM.getUsersInfo(userIDs);
const users = usersResponse.data ?? [];
const { data: friends } = await OpenIM.getFriendListPage({
  offset: 0,
  count: 50,
  filterBlack: true,
});
const { data: blocked } = await OpenIM.getBlackList();
```

Subscribe to presence only for users visible in the current UI.

## Returned data and events

The SDK response `data` field contains the updated OpenIM object or operation result. Persist IDs such as `conversationID`, `groupID`, and `clientMsgID` in your application state.

Watch these callbacks while testing: `CbEvents.OnSelfInfoUpdated`, `CbEvents.OnUserStatusChanged`, `CbEvents.OnFriendAdded`, `CbEvents.OnFriendDeleted`, `CbEvents.OnBlackAdded`, `CbEvents.OnBlackDeleted`.

Always log `operationID` with failures.

## Verify the result

- Confirm `login()` succeeds and `CbEvents.OnConnectSuccess` fires before calling the task API.
- Confirm the OpenIM call resolves and the returned data or follow-up SDK event matches the expected state.
- Confirm refresh and reconnect behavior matches the product requirement.

## Troubleshooting

| Case                           | Cause                                                                 | Handling                                                                                               |
| ------------------------------ | --------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| WASM asset requests return 404 | The SDK assets were not copied to the app public path.                | Copy `openIM.wasm`, `sql-wasm.wasm`, and `wasm_exec.js` from the npm package assets directory.         |
| The client never connects      | `apiAddr`, `wsAddr`, token, platform ID, or network routing is wrong. | Register connection events before `login()` and test the API and WebSocket URLs from the same browser. |
| The SDK call rejects | The server rejected the request or the current user lacks permission. | Handle the error in `catch`, log the rejected error with `operationID`, and verify group role, token, and target IDs. |
