页面刷新、浏览器重新激活或应用从离线状态恢复后，可以查询尚未处理的邀请和房间信息，再重建通话界面。查询结果应与实时通话事件统一合并处理，不能仅凭本地界面状态判断当前通话。

## 恢复启动时未处理的邀请

`getSignalingInvitationInfoStartApp()` 返回应用启动时需要恢复的邀请信息。该方法不接收请求参数。

```ts
const { data } =
  await OpenIM.getSignalingInvitationInfoStartApp();

if (data.invitation?.roomID) {
  restoreIncomingCall(data.invitation, data.offlinePushInfo);
}
```

恢复后仍需监听取消、超时、接受和挂断事件，因为读取结果只代表查询时刻的状态。

## 按群组查询房间

群聊通话可以使用 `signalingGetRoomByGroupID()` 查询 `CallingRoomData`。该方法的参数是群组 ID（`groupID`），不是自定义 `roomID`。通常建议在发起群聊通话时把 `RtcInvite.roomID` 设为同一个 `groupID`，这样发起、恢复和按群查询都使用同一标识；如果业务使用自定义 `roomID`，仍应用 `groupID` 调用本方法，另用保存的 `roomID` 去合并通话状态或调用 `signalingGetTokenByRoomID()`。

```ts
const { data: room } =
  await OpenIM.signalingGetRoomByGroupID(groupID);

if (room.roomID) {
  renderParticipants(room.participant ?? []);
}
```

返回的快照包含 `roomID`，并可能包含邀请和参与者信息。没有进行中的群聊通话时，服务端可能返回错误或空的房间数据。应用应把这种情况视为“当前无可恢复通话”，而不是继续使用旧的本地凭据。

## 获取房间凭据

已经知道 `roomID`、但需要重新取得入会凭据时，调用 `signalingGetTokenByRoomID()`。返回的 `RtcInviteResults` 包含 `token`、`liveURL` 和 `roomID`。

```ts
const { data: roomCredentials } =
  await OpenIM.signalingGetTokenByRoomID(roomID);

await mediaEngine.join({
  roomID: roomCredentials.roomID,
  token: roomCredentials.token,
  liveURL: roomCredentials.liveURL,
});
```

房间 token 属于短期敏感凭据。只在内存中交给媒体引擎，不要写入日志、URL、分析事件或持久化存储。

## 查询结果与事件增量

本页三个查询 API 的 Promise 成功后，`data` 分别是启动邀请、群组房间或房间凭据在当前时刻的快照。查询本身不会触发通话事件，也不需要等待事件后才能恢复界面或连接媒体引擎。

邀请生命周期、参与者和媒体流的后续变化通过通话事件增量更新，完整监听和清理代码见[发起或处理通话](/sdk/wasm/calling/managing-calls/start-or-handle-a-call)。房间状态按 `roomID` 合并，参与者状态同时使用用户 ID。连接恢复或本地状态与事件不一致时，重新查询房间信息；不要把某个事件当作此前查询 Promise 的完成回调。
