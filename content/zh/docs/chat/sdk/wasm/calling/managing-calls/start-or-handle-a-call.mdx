发起通话时，先构造 `RtcInvite`。单聊通话调用 `signalingInvite()`，群聊通话调用 `signalingInviteInGroup()`。两个方法都接收包含 `invitation` 和可选 `offlinePushInfo` 的对象。OpenIMClientSDK 负责邀请、接受、拒绝和挂断等信令状态。应用仍需使用返回的房间凭据，接入所选的实时音视频媒体引擎。

## 发起单聊通话

```ts
import { Platform, SessionType } from '@openim/wasm-client-sdk';

const invitation = {
  inviterUserID: currentUserID,
  inviteeUserIDList: ['user_b'],
  customData: JSON.stringify({ source: 'contact-card' }),
  groupID: '',
  roomID: crypto.randomUUID(),
  timeout: 30,
  mediaType: 'video',
  sessionType: SessionType.Single,
  platformID: Platform.Web,
};

const { data: roomCredentials } = await openimsdk.signalingInvite({
  invitation,
  offlinePushInfo: {
    title: '视频通话',
    desc: '你收到一个视频通话邀请',
    ex: '',
    iOSPushSound: 'default',
    iOSBadgeCount: true,
  },
});
```

### 参数说明

单聊邀请中的 `invitation` 字段含义如下：

| 字段 | 说明 |
| --- | --- |
| `inviterUserID` | 当前登录用户 ID，也就是本次通话的发起人。 |
| `inviteeUserIDList` | 被邀请用户 ID 列表。单聊只填写对方一个用户 ID，不要包含当前用户。 |
| `customData` | 可选的业务扩展字符串。需要传递结构化数据时先自行序列化，接收端按相同协议解析。 |
| `groupID` | 单聊没有群组，固定传空字符串。虽然语义为空，该字段在 `RtcInvite` 中仍为必填字段。 |
| `roomID` | 本次通话的房间标识。单聊应为每次新通话生成新的唯一值，后续事件和通话状态都按它合并。 |
| `timeout` | 邀请等待时长，单位为秒。超时后应关闭等待或来电界面。 |
| `mediaType` | 媒体类型字符串，例如 `audio` 或 `video`。这是业务约定值，发起端、接收端和媒体层必须使用相同约定。 |
| `sessionType` | 单聊传 `SessionType.Single`。 |
| `platformID` | 发起邀请的当前客户端平台；网页端传 `Platform.Web`，其他平台传对应枚举值。 |

`RtcInvite` 声明中还包含可选的 `initiateTime` 和 `busyLineUserIDList`。`initiateTime` 是以秒为单位的 Unix 时间戳，未传时由信令流程补充。`busyLineUserIDList` 是 OpenIMServer 返回的忙线结果，不应由发起端预先填写。

`offlinePushInfo` 是可选的离线推送内容，不属于 `RtcInvite`。其中 `title` 和 `desc` 是通知标题与正文，`ex` 可承载业务扩展字符串，`iOSPushSound` 和 `iOSBadgeCount` 控制 iOS 提示音与角标。是否能够收到离线推送，仍取决于 OpenIMServer 和客户端的推送配置。

Promise 成功后，`roomCredentials` 是 `RtcInviteResults`，包含 `roomID`、`token`、`liveURL`，以及可选的 `busyLineUserIDList`。取得有效的 `token` 和 `roomID` 后再连接媒体房间。Promise 成功不表示被邀请方已经接听。

## 发起群聊通话

群聊邀请只发送给 `inviteeUserIDList` 中明确列出的群成员，不会因为填写了 `groupID` 就自动邀请全群成员：

```ts
import { Platform, SessionType } from '@openim/wasm-client-sdk';

const groupInvitation = {
  inviterUserID: currentUserID,
  inviteeUserIDList: selectedGroupMemberIDs,
  customData: JSON.stringify({ source: 'group-call' }),
  groupID,
  roomID: groupID,
  timeout: 30,
  mediaType: 'video',
  sessionType: SessionType.WorkingGroup,
  platformID: Platform.Web,
};

const { data: roomCredentials } = await openimsdk.signalingInviteInGroup({
  invitation: groupInvitation,
  offlinePushInfo: {
    title: '群聊视频通话',
    desc: '你收到一个群聊视频通话邀请',
    ex: JSON.stringify({ groupID }),
    iOSPushSound: 'default',
    iOSBadgeCount: true,
  },
});
```

### 群聊字段差异

群聊邀请与单聊邀请使用同一个 `RtcInvite` 结构，但以下字段必须按群聊语义填写：

| 字段 | 说明 |
| --- | --- |
| `inviteeUserIDList` | 本次实际邀请的群成员 ID。先排除当前用户、空值和重复 ID；没有选择的群成员不会收到本次邀请。 |
| `groupID` | 目标群组 ID，不能传空字符串。发起前应确认当前用户仍在群内。 |
| `roomID` | 群通话房间标识。示例沿用 OpenIMSDK 的做法使用 `groupID`；如果业务自行生成房间 ID，必须保证本次群通话的所有参与端使用同一个值，并避免与仍在进行的房间冲突。 |
| `sessionType` | 群聊传 `SessionType.WorkingGroup`。不要沿用单聊的 `SessionType.Single`。 |

`inviterUserID`、`customData`、`timeout`、`mediaType` 和 `platformID` 与单聊含义一致。群成员选择、能否发起群通话等权限，需要客户端根据群资料和群成员字段在业务层处理。OpenIMServer 拒绝请求时，应直接展示接口返回的错误，不能把失败成员当成已进入房间。

`signalingInviteInGroup()` 的 Promise 成功同样返回 `RtcInviteResults`。`busyLineUserIDList` 只表示其中一部分成员忙线，不应因此中止其余成员的邀请和入会流程。

## 接受或拒绝邀请

被邀请方从 `CbEvents.OnReceiveNewInvitation` 取得 `RtcInvite`。用户确认接听后调用 `signalingAccept()`；拒绝时调用 `signalingReject()`。

```ts
const action = {
  opUserID: currentUserID,
  invitation,
};

const { data: roomCredentials } = await openimsdk.signalingAccept(action);

// 用户拒绝时：
await openimsdk.signalingReject(action);
```

`opUserID` 是执行接受、拒绝、取消或挂断操作的当前用户 ID。不要用邀请人 ID 固定填充该字段。

`invitation` 必须使用收到的原始 `RtcInvite`，尤其要保留同一个 `roomID`、邀请人、被邀请人列表和会话类型，不要在接受或拒绝时重新生成房间 ID。

`signalingAccept()` 成功返回 `RtcInviteResults`；接听端取得有效 `token` 和 `roomID` 后再连接媒体房间。`signalingReject()` 成功只表示 OpenIMServer 已接受拒绝请求，发起端界面随后由 `OnInviteeRejected` 更新。

## 取消或挂断

邀请尚未接通时，发起方调用 `signalingCancel()`；通话已经建立后，参与者调用 `signalingHungUp()`。

```ts
await openimsdk.signalingCancel({ opUserID: currentUserID, invitation });

await openimsdk.signalingHungUp({ opUserID: currentUserID, invitation });
```

这些调用只更新 OpenIMClientSDK 信令状态。应用还需要停止本地采集、断开媒体房间，并释放摄像头和麦克风资源。为避免重复操作，建议按 `roomID` 维护通话状态机，并让取消、拒绝、超时和挂断都进入同一个清理流程。

`signalingCancel()` 和 `signalingHungUp()` 都接收当前用户的 `opUserID` 与本次通话的原始 `invitation`。前者只能用于尚未接通的邀请，后者用于已经建立的通话。不要通过重新构造一个只有 `roomID` 的对象代替完整邀请。

## 处理忙线和失败

调用失败时记录 `errCode`、`errMsg` 和当前会话标识。在获得有效 `token` 和 `roomID` 之前，不要启动媒体房间连接。

`signalingReject()`、`signalingCancel()` 和 `signalingHungUp()` 的 Promise 成功表示 OpenIMServer 已受理并完成当前信令请求；远端界面是否更新仍由对应事件决定。

## 监听通话状态

通话界面需要同时处理邀请生命周期、参与者连接状态和媒体流变化。进入通话页面或初始化通话模块前注册事件，按 `roomID` 合并到同一份本地状态。

| 事件 | 应用处理 |
| --- | --- |
| `OnReceiveNewInvitation` | 展示新的来电邀请，并启动超时计时。 |
| `OnInviteeAccepted` | 标记被邀请方已接受，准备或继续媒体连接。 |
| `OnInviteeRejected` | 标记被邀请方已拒绝。 |
| `OnInvitationCancelled` | 关闭尚未接通的来电界面。 |
| `OnInvitationTimeout` | 结束等待状态并显示未接听结果。 |
| `OnInviteeAcceptedByOtherDevice` | 当前设备停止响铃，由已接受邀请的其他设备继续处理。 |
| `OnInviteeRejectedByOtherDevice` | 当前设备停止响铃，并同步其他设备的拒绝结果。 |
| `OnHangUp` | 结束通话并释放媒体资源。 |
| `OnRoomParticipantConnected` | 合并新加入房间的参与者。 |
| `OnRoomParticipantDisconnected` | 移除离开房间的参与者。 |
| `OnStreamChange` | 更新参与者的媒体流状态。 |

```ts
import { CbEvents } from '@openim/wasm-client-sdk';

const handlers = {
  invitation: ({ data }) => upsertCall({ invitation: data, state: 'ringing' }),
  accepted: ({ data }) => mergeCallEvent('accepted', data),
  rejected: ({ data }) => mergeCallEvent('rejected', data),
  cancelled: ({ data }) => mergeCallEvent('cancelled', data),
  timeout: ({ data }) => mergeCallEvent('timeout', data),
  acceptedByOtherDevice: ({ data }) => {
    mergeCallEvent('accepted-by-other-device', data);
  },
  rejectedByOtherDevice: ({ data }) => {
    mergeCallEvent('rejected-by-other-device', data);
  },
  hangUp: ({ data }) => finishCall(getCallRoomID(data)),
  participantConnected: ({ data }) => addParticipant(data),
  participantDisconnected: ({ data }) => removeParticipant(data),
  streamChanged: ({ data }) => updateParticipantStream(data),
};

openimsdk.on(CbEvents.OnReceiveNewInvitation, handlers.invitation);
openimsdk.on(CbEvents.OnInviteeAccepted, handlers.accepted);
openimsdk.on(CbEvents.OnInviteeRejected, handlers.rejected);
openimsdk.on(CbEvents.OnInvitationCancelled, handlers.cancelled);
openimsdk.on(CbEvents.OnInvitationTimeout, handlers.timeout);
openimsdk.on(CbEvents.OnInviteeAcceptedByOtherDevice, handlers.acceptedByOtherDevice);
openimsdk.on(CbEvents.OnInviteeRejectedByOtherDevice, handlers.rejectedByOtherDevice);
openimsdk.on(CbEvents.OnHangUp, handlers.hangUp);
openimsdk.on(CbEvents.OnRoomParticipantConnected, handlers.participantConnected);
openimsdk.on(CbEvents.OnRoomParticipantDisconnected, handlers.participantDisconnected);
openimsdk.on(CbEvents.OnStreamChange, handlers.streamChanged);

function removeCallListeners() {
  openimsdk.off(CbEvents.OnReceiveNewInvitation, handlers.invitation);
  openimsdk.off(CbEvents.OnInviteeAccepted, handlers.accepted);
  openimsdk.off(CbEvents.OnInviteeRejected, handlers.rejected);
  openimsdk.off(CbEvents.OnInvitationCancelled, handlers.cancelled);
  openimsdk.off(CbEvents.OnInvitationTimeout, handlers.timeout);
  openimsdk.off(CbEvents.OnInviteeAcceptedByOtherDevice, handlers.acceptedByOtherDevice);
  openimsdk.off(CbEvents.OnInviteeRejectedByOtherDevice, handlers.rejectedByOtherDevice);
  openimsdk.off(CbEvents.OnHangUp, handlers.hangUp);
  openimsdk.off(CbEvents.OnRoomParticipantConnected, handlers.participantConnected);
  openimsdk.off(CbEvents.OnRoomParticipantDisconnected, handlers.participantDisconnected);
  openimsdk.off(CbEvents.OnStreamChange, handlers.streamChanged);
}
```

`OnReceiveNewInvitation` 的 `data` 是 `RtcInvite`。其余邀请生命周期事件的 `data` 是对应信令状态载荷，先从载荷中的邀请或房间字段解析出 `roomID`。参与者连接、断开和媒体流事件还要结合用户 ID 更新参与者状态。不要按事件到达顺序或数组当前位置合并状态。

本页是以上 11 个通话事件的完整监听示例归属页。离开通话功能、退出登录或切换账号时调用 `removeCallListeners()`。按 `roomID` 定位通话，并结合用户 ID 更新参与者状态。

取消、拒绝、超时和挂断都应进入同一个资源清理流程。需要显示当前房间快照时，再查询房间信息。
