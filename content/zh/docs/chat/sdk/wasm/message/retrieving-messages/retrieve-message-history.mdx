OpenIMClientSDK 使用 `conversationID` 和 `clientMsgID` 游标读取历史消息。进入会话时先请求第一页；用户上拉或滚动到边界时，再把上一页边界消息的 `clientMsgID` 作为下一次请求的 `startClientMsgID`。

OpenIMSDK 的历史消息读取以会话为中心。单聊和群聊使用同一套分页模型；根据群组获取对应会话时使用 `SessionType.WorkingGroup`，无需维护另一套消息列表接口。

## 按会话读取消息

常规聊天页优先使用 `getAdvancedHistoryMessageList()`。第一页没有游标时，把 `startClientMsgID` 设为空字符串；继续加载时，使用当前历史列表边界消息的 `clientMsgID`。

```ts
import { SessionType, ViewType } from '@openim/wasm-client-sdk';
import type { MessageItem } from '@openim/wasm-client-sdk';

type HistoryPage = {
  messages: MessageItem[];
  nextCursor: string;
  isEnd: boolean;
};

const loadHistoryPage = async (
  conversationID: string,
  startClientMsgID = '',
): Promise<HistoryPage> => {
  const { data } = await openimsdk.getAdvancedHistoryMessageList(
    {
      conversationID,
      startClientMsgID,
      count: 30,
      viewType: ViewType.History,
    }
  );

  return {
    messages: data.messageList,
    nextCursor: data.messageList.at(-1)?.clientMsgID ?? '',
    isEnd: data.isEnd,
  };
};
```

调用方可以把第一页用于初始化消息列表，将后续页插入消息列表较早的一端。合并消息时建议以 `clientMsgID` 去重，避免登录同步、新消息事件和历史分页返回同一条消息。

```ts
let historyCursor = '';
let reachedHistoryEnd = false;

async function loadOlderMessages(conversationID: string) {
  if (reachedHistoryEnd) {
    return;
  }

  const page = await loadHistoryPage(conversationID, historyCursor);

  prependMessages(page.messages);
  historyCursor = page.nextCursor;
  reachedHistoryEnd = page.isEnd || !page.nextCursor;
}
```

### 参数说明

| 参数               | 类型       | 是否必填 | 说明                                                                                                              |
| ------------------ | ---------- | -------- | ----------------------------------------------------------------------------------------------------------------- |
| `conversationID`   | `string`   | 必填     | 要读取历史消息的会话 ID。它来自会话列表、会话事件，或通过 `getConversationIDBySessionType()` 根据用户或群组换取。 |
| `startClientMsgID` | `string`   | 必填     | 分页游标。第一页传空字符串；后续请求传入上一页边界消息的 `clientMsgID`。                                          |
| `count`            | `number`   | 必填     | 每页消息数量。聊天页常用 20 到 50 条，避免一次性渲染过多历史消息。                                                |
| `viewType`         | `ViewType` | 必填     | 读取视图类型。历史列表使用 `ViewType.History`。                                                     |

返回值中的 `data.messageList` 是本页消息，`data.isEnd` 表示是否已经到达历史边界。

```ts
type AdvancedGetMessageResult = {
  isEnd: boolean;
  errCode: number;
  errMsg: string;
  messageList: MessageItem[];
};
```

历史查询的 Promise 成功后，`data` 就是本次请求的消息快照。读取历史不会触发新消息事件，也不需要等待事件才能渲染；同一条消息可能同时出现在分页结果和接收事件中，因此应结合目标 `conversationID` 并按 `clientMsgID` 幂等合并。

## 群聊消息列表

如果业务入口只有 `groupID`，先取得群组对应的 `conversationID`，再使用同一个历史消息接口读取列表。

```ts
const { data: conversationID } = await openimsdk.getConversationIDBySessionType({
  sourceID: groupID,
  sessionType: SessionType.WorkingGroup,
});

const { data: firstPage } = await openimsdk.getAdvancedHistoryMessageList({
  conversationID,
  startClientMsgID: '',
  count: 30,
  viewType: ViewType.History,
});
```

也可以通过 `getOneConversation()` 读取群会话状态，再使用返回的 `conversationID` 打开聊天页。

```ts
const { data: conversation } = await openimsdk.getOneConversation({
  sourceID: groupID,
  sessionType: SessionType.WorkingGroup,
});

openGroupMessageList(conversation.conversationID);
```

`getConversationIDBySessionType()` 和 `getOneConversation()` 都直接通过 Promise 返回当前结果，查询本身不会触发会话或消息事件。群聊页面打开期间，新消息不需要等下一次历史分页，可以直接通过[接收消息](/sdk/wasm/message/receiving-messages/receive-messages)中的通用事件处理器合并；本页只负责历史分页，不重复注册新消息或同步事件。

## 选择读取方向

`getAdvancedHistoryMessageList()` 和 `getAdvancedHistoryMessageListReverse()` 使用同一组参数，适合已经持有 `conversationID` 的聊天页。两者返回的都是 `AdvancedGetMessageResult`，包含 `messageList` 和 `isEnd`。

```ts
const { data: result } = await openimsdk.getAdvancedHistoryMessageListReverse({
  conversationID,
  startClientMsgID: cursorClientMsgID,
  count: 30,
  viewType: ViewType.History,
});
```

如果调用方只有单聊用户 ID 或群组 ID，也可以使用 `getHistoryMessageListReverse()`。该方法仍建议传入 `conversationID`；单聊时填写 `userID`，群聊时填写 `groupID`，另一侧目标字段保持为空字符串。

```ts
const { data: groupHistory } = await openimsdk.getHistoryMessageListReverse({
  userID: '',
  groupID,
  conversationID,
  startClientMsgID: cursorClientMsgID,
  count: 30,
});
```

新页面通常不需要在高级历史接口与传统历史接口之间混用。统一一种读取方向和一种游标规则，可以减少重复消息、顺序反转和滚动位置跳动。

## 游标和空列表

当 `messageList` 为空且 `isEnd` 为 `true` 时，可以停止继续请求历史页。当 `messageList` 为空但界面仍需要等待同步完成时，先保留当前游标，并在[事件概览](/sdk/wasm/events/overview-events)说明的同步完成事件后重新尝试当前会话。

不要用数组下标作为分页游标。消息撤回、删除、本地同步和多端登录都可能改变当前列表长度；`clientMsgID` 才是历史读取和去重应使用的稳定标识。

需要刷新当前消息列表时，可以保留当前列表并重新读取边界页。重新查询、事件增量和此前 Promise 的完成互不替代；这三条路径都应按相同的会话标识和 `clientMsgID` 合并消息。重新登录后的消息变化由事件同步。
