OpenIMSDK 的入群申请由群主或管理员处理。它和 `inviteUserToGroup()` 的成员邀请不同：申请人主动提交入群请求后，由有权限的群成员审核；成员邀请则由群内有权限的用户发起。

## 查询申请

管理端使用 `getGroupApplicationListAsRecipient()` 获取收到的申请，申请人使用 `getGroupApplicationListAsApplicant()` 获取自己发出的申请。两个方法使用相同的筛选和分页对象，只是查询方向不同。

### 参数说明

| 参数 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| `groupID` | `string[]` | 必填 | 要查询的群组 ID 列表。 |
| `handleResults` | `number[]` | 必填 | 要查询的申请处理状态。 |
| `offset` | `number` | 必填 | 起始偏移量，第一页传 `0`。 |
| `count` | `number` | 必填 | 本次读取数量。 |

### 查询收到的申请

```ts
import { ApplicationHandleResult } from '@openim/wasm-client-sdk';

const { data: pendingApplications } = await openimsdk.getGroupApplicationListAsRecipient({
  groupID: [groupID],
  handleResults: [ApplicationHandleResult.Unprocessed],
  offset: 0,
  count: 20,
});
```

`ApplicationHandleResult` 中 `Unprocessed` 表示未处理，`Agree` 表示已同意，`Reject` 表示已拒绝。不要使用不存在的状态值 `2`。

Promise 成功后，`data` 是当前用户有权处理的 `GroupApplicationItem[]`。

### 查询发出的申请

申请人查询自己提交的记录时，使用相同的群组、状态和分页字段：

```ts
const { data: sentApplications } = await openimsdk.getGroupApplicationListAsApplicant({
  groupID: [groupID],
  handleResults: [
    ApplicationHandleResult.Unprocessed,
    ApplicationHandleResult.Agree,
    ApplicationHandleResult.Reject,
  ],
  offset: 0,
  count: 20,
});
```

Promise 成功后，`data` 是当前用户发出的 `GroupApplicationItem[]`。

## 未处理数量

### 按时间统计未处理申请

`getGroupApplicationUnhandledCount({ time })` 返回当前用户作为群主或管理员需要处理的申请数量。

```ts
const { data: count } = await openimsdk.getGroupApplicationUnhandledCount({
  time: 0,
});
```

`time` 是统计起点时间。Promise 成功后，`data` 是该时间点之后需要当前用户处理的申请数量。

### 获取申请角标

`getGroupApplicationBadgeCount()` 返回用于申请入口角标的计数。它和按时间统计的未处理申请数是两个独立入口：

```ts
const { data: badgeCount } = await openimsdk.getGroupApplicationBadgeCount();
updateGroupApplicationBadge(badgeCount);
```

### 清除申请角标

用户打开并确认申请列表后，可以清除角标：

```ts
await openimsdk.clearGroupApplicationBadgeCount();
```

清除角标不会同意、拒绝或删除申请。角标变化通过 `CbEvents.OnGroupApplicationBadgeCountChanged` 同步，事件的 `data` 是最新计数。

## 同意或拒绝

申请记录使用 `userID` 表示申请人，处理接口把同一个值命名为 `fromUserID`。

### 参数说明

| 参数 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| `groupID` | `string` | 必填 | 申请对应的群组 ID。 |
| `fromUserID` | `string` | 必填 | 申请人的用户 ID，对应申请记录中的 `userID`。 |
| `handleMsg` | `string` | 必填 | 同意或拒绝时的处理说明；没有说明时传空字符串。 |

```ts
await openimsdk.acceptGroupApplication({
  groupID: application.groupID,
  fromUserID: application.userID,
  handleMsg: '同意加入',
});

await openimsdk.refuseGroupApplication({
  groupID: application.groupID,
  fromUserID: application.userID,
  handleMsg: '暂不接受该申请',
});
```

列表状态通过群申请事件和重新查询同步。处理成功不应直接伪造最终 `GroupMemberItem`。

## 删除申请记录

`deleteGroupRequests()` 按申请人和群组的组合批量删除申请记录：

```ts
await openimsdk.deleteGroupRequests([
  {
    fromUserID: application.userID,
    groupID: application.groupID,
  },
]);
```

删除记录不等于拒绝待处理申请。需要表达审核结果时，应先使用 `acceptGroupApplication()` 或 `refuseGroupApplication()`；删除只用于产品明确提供的记录清理入口。

## 调用后的状态变化

查询申请列表、未处理数量或角标计数时，应直接使用 Promise 返回值建立当前快照；这些查询不会因为读取数据而触发申请事件。

会改变申请状态的调用需要分别处理请求结果和事件增量：

- `acceptGroupApplication()` 的 Promise 成功表示 OpenIMServer 已完成本次同意请求，随后通过 `OnGroupApplicationAccepted` 合并已同意的申请记录。
- `refuseGroupApplication()` 的 Promise 成功表示 OpenIMServer 已完成本次拒绝请求，随后通过 `OnGroupApplicationRejected` 合并已拒绝的申请记录。
- `deleteGroupRequests()` 的 Promise 成功表示删除请求已完成，随后通过 `OnGroupApplicationDeleted` 移除对应记录。
- `clearGroupApplicationBadgeCount()` 的 Promise 成功表示清除请求已完成，最新角标以 `OnGroupApplicationBadgeCountChanged` 的数值载荷为准。

`OnGroupApplicationAdded` 表示收到了新的入群申请，不是查询申请列表产生的回调。申请事件的 `data` 是 `GroupApplicationItem`；合并或删除记录时使用申请人 `userID` 与 `groupID` 的组合作为标识，不要依赖列表位置。首次进入页面或用户主动刷新时，再重新查询列表和计数；重新登录后的变化由事件同步。

如果申请列表仍在继续分页，新增、处理或删除事件可能改变后续记录的位置。收到这些事件后应重置分页状态并从第一页重新加载，再继续翻页。

## 保持申请列表同步

入群申请的新增、同意、拒绝和删除分别通过对应事件同步；角标变化使用 `OnGroupApplicationBadgeCountChanged`：

```ts
import { CbEvents } from '@openim/wasm-client-sdk';

const handlers = {
  added: ({ data }) => mergeGroupApplication(data),
  accepted: ({ data }) => mergeGroupApplication(data),
  rejected: ({ data }) => mergeGroupApplication(data),
  deleted: ({ data }) => removeGroupApplication(data),
  badgeChanged: ({ data }) => updateGroupApplicationBadge(data),
};

openimsdk.on(CbEvents.OnGroupApplicationAdded, handlers.added);
openimsdk.on(CbEvents.OnGroupApplicationAccepted, handlers.accepted);
openimsdk.on(CbEvents.OnGroupApplicationRejected, handlers.rejected);
openimsdk.on(CbEvents.OnGroupApplicationDeleted, handlers.deleted);
openimsdk.on(CbEvents.OnGroupApplicationBadgeCountChanged, handlers.badgeChanged);

function removeGroupApplicationListeners() {
  openimsdk.off(CbEvents.OnGroupApplicationAdded, handlers.added);
  openimsdk.off(CbEvents.OnGroupApplicationAccepted, handlers.accepted);
  openimsdk.off(CbEvents.OnGroupApplicationRejected, handlers.rejected);
  openimsdk.off(CbEvents.OnGroupApplicationDeleted, handlers.deleted);
  openimsdk.off(CbEvents.OnGroupApplicationBadgeCountChanged, handlers.badgeChanged);
}
```

事件可能在主动处理申请后再次到达，应按申请记录的用户和群组标识幂等合并。页面卸载或监听作用域结束时调用 `removeGroupApplicationListeners()`，避免重复注册。
